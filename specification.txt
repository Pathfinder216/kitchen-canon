What I want this app to do:
- I want to be able to add recipes from any source (a food blog webpage, text from an email, a PDF or docx file, a picture of an index card, etc.). Preferably this process is as automated as possible. The original source should be linked/named, if applicable. These will automatically be saved in my personal set of recipes.
- I want to be able to add recipe author notes that are separate from the ingredients/steps (e.g. specifying what pan size to use when doubling the recipe).
- I want to be able to easily follow the recipe, edit it, and add personal notes for the future (e.g. changes I want to try but am not confident in yet, so I don't want to edit the recipe) as I cook/bake the recipe.
- I want to be able to share the recipe with others such that they can access it in a way that is convenient for them (raw text via email, PDF via email, link to the recipe on my app via text, etc.).
- I want ingredient names to be standardized (e.g. a British recipe calling for coriander would be translated into cilantro for an American user). This will be different based on the user's localization.
- I want to be able to convert ingredients to common substitutes (e.g. if a recipe calls for fresh thyme, I can substitute dried thyme in, and the ingredient list will show that substitution).
    - This will be possible when viewing or editing a recipe or when planning or cooking a meal.
    - If there are multiple substitute possibilities, I want to be able to select which one I want.
    - The substitution data can be built up over time by users (e.g. a user can add a substitution for eggs, providing a vegan option).
- I want to be able to mark ingredients as optional (e.g. mushrooms in a stir fry are optional).
- I want to be able to specify time required for individual steps of the recipe, and whether it is active or inactive time.
- I want to be able to publish my recipes for other users to view/discover and save.
- I want to be able to view and search my recipes.
    - I can search by text in the recipe title.
    - I can filter for recipes that use one or more particular ingredients.
    - I can filter out recipes that use particular ingredients.
    - I can filter for particular labels/categories.
- I want recipes to be automatically labelled for dietary restrictions and allergens based on the ingredients (e.g. a recipe containing only eggs, salt, and pepper is labelled gluten-free). I also want it to be easy to tell if the recipe can easily be made to match a dietary restriction (e.g. a non-gluten-free recipe can be made gluten free by using a one-for-one gluten-free flour instead of regular all purpose flour). These labels can be edited manually.
- I want recipes to be categorized in all reasonable ways (one recipe is a dinner entree, another recipe is a dinner entree or an appetizer, another recipe is a spicy side dish, etc.).
- I want to be able to label/categorize recipes based on whether they can be made ahead of time and whether they need to be refrigerated/frozen. For example, I might want to make a dinner, but I'll know I have little to no time to cook it before eating/serving it, so I'd want to find a recipe that I can make the day before. For another example, I might be going somewhere right before I take dinner somewhere, and I won't be able to refrigerate the dish for a couple hours, so I don't want to have to refrigerate it.
- I also want to be able to label/categorize a recipe as using a slow cooker.
- The above labelling/categorization should be optional, so I'm not forced to spend a bunch of time labelling a recipe I'm adding when I might not know much of anything about it yet.
- I want to be able to record a video or include a pre-existing video (especially videos from food blog webpages).
- I want to be able to include images. I'm mostly thinking an image of the finished product and an image for each step.
- I don't want the video and images, if they are included, to get in the way. Sometimes I don't care about them and I just want the recipe step text without other things clogging the screen space.
- I want the number of servings that are made to be shown and arbitrarily adjustable so the recipe can be made for any number of servings. For now, this will just affect ingredient amounts.
- I want to be able to specify in the recipe steps what percent amount of an ingredient is being used and have it automatically display the non-percent amount based on the selected number of servings/ingredient amount (e.g. if a recipe calls for 2 Tbsp of oil and the user doubled the serving size, a step can specify using half of the oil, and it will show "2 Tbsp" of oil).
- I want the amount of time needed to made the recipe listed. I want to know both the total time needed from starting the recipe to finishing the recipe and the amount of active time needed (e.g. cutting vegetables and stirring is active time and the dish just sitting in the oven is not active time).
- I want to be able to plan a full meal including multiple recipes.
    - I can specify dietary restrictions/preferences (e.g. gluten free and not spicy and no mushrooms) to filter out recipes that I can't use.
    - I can easily pick multiple recipes. This might look something like adding items to a shopping cart. In the future, I want to make it easy to pick complementary recipes (e.g. if I pick a salad entree, I probably don't want to be scrolling through salads when picking a side dish); design/plan/implement with this in mind.
    - After picking my recipes, create a consolidated grocery list. This should be easily editable to remove things I already have and be easy to copy into some other app.
    - In the long term, I would like to automatically generate a plan for how long before serving I should start cooking and what order I should make the recipes in. Even better would be being able to work on recipes concurrently (e.g. I start the next recipe after putting a dish in the oven). It would also be good to be able to separate things that can be made ahead of time (e.g. a refrigerated dessert is made on its own the night before). Keep this in mind when designing/architecting/implementing this app.
- I want a cook mode where I can cook my planned full meal, which will easily allow me to switch between the selected recipes and will hide other unnecessary parts of the app.
- I want there to be a history of my planned full meals that I can view.
- I want to be able to remake a previous planned full meal (i.e. create/show the grocery list again and allow me to go into cook mode for that meal again).
- I want to be able to prevent my phone screen from going dark while following the recipe/cooking a meal.
- I want to be able to use this on my phone without internet access. The only feature I want to lose when without internet access is discovering other users' recipes.
- I want to be able to export all my recipes in a standardized format.
    - If it's not possible to do this without losing any data/labels added in my app, I also want to be able to export in a proprietary format that retains all of this data.
- When following a recipe, I want to be able to tap a button or swipe to proceed to the next step in the recipe.
- I want to be able to archive my recipes so they don't show up when searching. They will still exist in the meal history, if applicable, and they can be unarchived.
- I want recipes to be versioned (i.e. every time a recipe is edited, a new version of that recipe is saved).
- I want users to be able to view/restore old versions of a recipe.
    - When viewing meal history, the recipes will be the version used in that meal.


Other notes:
- I will use my phone to do pretty much everything I would use it for (adding a recipe, following a recipe when cooking, sharing a recipe, etc.). I will also use my computer for at least some tasks, like adding and sharing recipes.
- I'm mostly concerned with making this for my own use, but if it turns out well, I may consider making it more publicly available. I am using a public GitHub repository for this project so others can use the source code at least.
- This app will run on my local Raspberry Pi for now, and its SD card will be the data storage. I may want to publish publically/scale in the future, so I want the data API abstracted so it will be easy to swap to using something like AWS or GCP or some other cloud storage provider.
- The initial version will not be a multi-user platform with recipe sharing. This is a stretch goal that should be kept in mind when designing/architecting/implementing the initial version.
    - Unaddressed concerns with this platform (e.g. privacy controls, accounts/permission/copyright concerns) will be addressed at a later time.
